# Planification de séance + notifications push — Design

**Date :** 2026-06-02
**Statut :** validé (forme) → prêt pour plan d'implémentation

## 1. Objectif

Permettre à l'utilisateur de **planifier sa semaine à l'avance** (un type de séance par jour) et de **recevoir le jour J une notification push** (même app fermée) qui l'invite à démarrer. À l'ouverture, la séance prévue est **mise en avant et pré-configurée** pour réduire la friction — **sans entraver** l'accès au reste de l'app ni empêcher de faire une autre séance.

C'est la **fondation** d'un système qui deviendra plus tard « intelligent » (suggestions, récurrence). On reste donc volontairement minimal.

## 2. Périmètre

**Dans le périmètre :**
- Un plan = `(date, type)`, un type par jour. Types = les 6 de `WORKOUT_TYPES` + `athletics`.
- Point d'entrée : une **chip dans le `StatusSpot`** (rangée d'actions du hero), intégrée au défilé existant.
- **Modal de planif minimal** : choisir un type par jour de la semaine, enregistrement en batch.
- **Jour J** : le CTA du bas de l'`IdleScreen` devient un **split-button** « Commencer {type} » (→ chrono direct, type pré-rempli) + une flèche révélant **un seul** autre choix : « Séance libre ».
- **Web Push réel** : abonnement, table d'abonnements, handlers SW, envoi serveur.
- **Planificateur** : Vercel Cron quotidien (~8h Europe/Paris) qui envoie le rappel du jour.

**Hors périmètre (évolutions futures) :**
- Templates d'exercices pré-remplis (l'option « B » écartée).
- Récurrence hebdomadaire automatique.
- Heure de rappel réglable par l'utilisateur / rappels multiples / snooze.
- Suggestions « intelligentes ».

## 3. Décisions d'architecture

- **Plan = type seul** (pas d'exos), cohérent avec le flux actuel où les exos se choisissent le jour J dans `exercise_select`.
- **Réutilisation du `StatusSpot`** plutôt qu'un nouveau carrousel : la planif rejoint le tableau `items[]` qui défile déjà. Aucun badge « ! » pulsant (réservé aux vraies alertes poids/profil).
- **Planificateur = Vercel Cron** (et non Supabase pg_cron) :
  - Hobby autorise 1 cron/jour, déclenché dans le créneau d'1h → suffisant pour un rappel matinal.
  - Bonus : la requête DB quotidienne du cron **empêche l'auto-pause Supabase free** (pause après 7 j sans requête). `pg_cron` au contraire meurt avec le projet en pause.
- **Batch-save** du plan (comme l'athlé) : le modal édite en mémoire, un seul appel au clic « Enregistrer ».

## 4. Modèle de données

Migration SQL manuelle (convention du projet), fichier `supabase/migrations/20260602_planning_push.sql`.

```sql
-- Séances planifiées : 1 type par jour et par utilisateur.
CREATE TABLE IF NOT EXISTS public.planned_sessions (
  id         bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  user_id    text NOT NULL DEFAULT (auth.jwt() ->> 'sub'),
  date       date NOT NULL,
  type       text NOT NULL CHECK (type IN ('push','pull','legs','full','upper','core','athletics')),
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now(),
  UNIQUE (user_id, date)
);

ALTER TABLE public.planned_sessions ENABLE ROW LEVEL SECURITY;
CREATE POLICY planned_select_own ON public.planned_sessions
  FOR SELECT USING (user_id = (auth.jwt() ->> 'sub'));
CREATE POLICY planned_insert_own ON public.planned_sessions
  FOR INSERT WITH CHECK (user_id = (auth.jwt() ->> 'sub'));
CREATE POLICY planned_update_own ON public.planned_sessions
  FOR UPDATE USING (user_id = (auth.jwt() ->> 'sub'))
             WITH CHECK (user_id = (auth.jwt() ->> 'sub'));
CREATE POLICY planned_delete_own ON public.planned_sessions
  FOR DELETE USING (user_id = (auth.jwt() ->> 'sub'));

-- Abonnements Web Push (1 par appareil/navigateur).
CREATE TABLE IF NOT EXISTS public.push_subscriptions (
  id         bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  user_id    text NOT NULL DEFAULT (auth.jwt() ->> 'sub'),
  endpoint   text NOT NULL UNIQUE,
  p256dh     text NOT NULL,
  auth       text NOT NULL,
  created_at timestamptz NOT NULL DEFAULT now()
);

ALTER TABLE public.push_subscriptions ENABLE ROW LEVEL SECURITY;
CREATE POLICY push_select_own ON public.push_subscriptions
  FOR SELECT USING (user_id = (auth.jwt() ->> 'sub'));
CREATE POLICY push_insert_own ON public.push_subscriptions
  FOR INSERT WITH CHECK (user_id = (auth.jwt() ->> 'sub'));
CREATE POLICY push_delete_own ON public.push_subscriptions
  FOR DELETE USING (user_id = (auth.jwt() ->> 'sub'));
```

> Le cron lit **tous** les `planned_sessions` du jour + leurs `push_subscriptions` : il utilise un **client admin** (clé secrète Supabase) qui bypasse la RLS. Aucune policy SELECT « globale » n'est créée.

## 5. Composants & flux

### 5.1 Données client — `usePlan`
Nouveau hook via `createCachedResource` (clé `tcp:plan:v1`, variante = lundi de la semaine, `staleMs` 3 min).
- `GET /api/plan?weekStart=YYYY-MM-DD` → `[{ date, type }]`.
- Helper dérivé `plannedToday(plans)` → l'entrée dont `date === aujourd'hui` (ou `null`).
- `invalidate()` après enregistrement du modal (purge SWR, comme `useExos`/`useProfile`).

### 5.2 Entrée — `StatusSpot` (`app/seance/_components/StatusSpot.tsx`)
Le tableau `items[]` (qui défile déjà toutes ~3,8 s) intègre la planif :
- **Sans séance ce jour** : `["Prêt à reprendre ? · {greeting}", "Planifier ma semaine"]` (+ alertes poids/profil existantes en tête si présentes).
- **Avec séance ce jour** : on **retire le greeting** → `["Aujourd'hui : {label}", "Ma semaine"]` (+ alertes).
- La chip planif ouvre `PlanWeekModal`. Couleur discrète (violet `--brand` pour « Planifier », teinte discipline pour « Aujourd'hui : … »). **Pas** de badge « ! » pulsant.

### 5.3 Modal — `PlanWeekModal` (`app/seance/_components/PlanWeekModal.tsx`)
- Liste Lun→Dim de la semaine courante. Chaque jour : libellé + date + pill du type assigné (vert muscu / orange athlé) ou « + Ajouter ».
- Taper un jour ouvre le choix de type : 6 types + Athlétisme + « Aucune » (retire le jour).
- Édition **en mémoire** ; bouton **Enregistrer** → un seul appel batch.
- `POST /api/plan` body `{ weekStart, entries: [{ date, type }] }` : upsert des jours présents, **suppression** des jours de cette semaine absents du payload. Puis `invalidate()`.
- **Après le tout premier enregistrement** : déclenche la demande de permission + abonnement push (§5.5). Si refusé/non supporté → le plan reste valide, simplement pas de notif.

### 5.4 Jour J — CTA `IdleScreen` (`app/seance/_screens/IdleScreen.tsx`)
Quand `plannedToday` existe, la barre fixe du bas devient un **split-button** :
- **Zone principale** « Commencer {label} » → `nav('config', { plannedType, configStep: 'chrono' })`.
- **Flèche `▾`** (segment ≥44px, `aria-haspopup`/`aria-expanded`) → révèle, en glissant vers le haut depuis le bouton, **un seul** item « Séance libre » → `nav('config')` (étape type, rien de présélectionné). Fermable tap-dehors / Échap. `prefers-reduced-motion` respecté.
- Sans `plannedToday` : CTA inchangé (« Commencer une séance »).

### 5.5 Pré-sélection — `SessionClient` + `ConfigScreen`
- `NavContext` gagne `plannedType?: string`.
- `SessionClient.nav` : si `plannedType` présent → `setSession((s) => ({ ...s, type: plannedType }))` et `setConfigInitialStep('chrono')`.
- `ConfigScreen` initialise déjà `type` depuis `session.type` et `step` depuis `initialStep` → s'ouvre sur le chrono avec le bon type. Retour « ← » depuis chrono ramène au sélecteur de type (override possible). Aucun changement de logique de fond.

### 5.6 PWA Push
- **`public/sw.js`** : ajouter handlers `push` (→ `showNotification(title, { body, icon, data:{ url:'/seance' } })`) et `notificationclick` (→ focus un onglet existant sinon `clients.openWindow('/seance')`). Bump `CACHE_VERSION`/`RUNTIME` → `v4`.
- **`app/seance/_lib/push.ts`** : `subscribeToPush()` — vérifie le support (`'serviceWorker' in navigator && 'PushManager' in window && 'Notification' in window`), demande la permission, `registration.pushManager.subscribe({ userVisibleOnly:true, applicationServerKey })`, `POST /api/push/subscribe`.
- **Garde iOS** : sur iOS, le push n'existe que si la PWA est **installée** (standalone). Si non standalone (`!window.matchMedia('(display-mode: standalone)').matches && !navigator.standalone`), on n'affiche pas la demande mais un **indice** « Ajoute à l'écran d'accueil pour activer les rappels ».

### 5.7 API
- `GET /api/plan?weekStart=` — liste de la semaine (client RLS via token Clerk).
- `POST /api/plan` — batch upsert/delete de la semaine (client RLS).
- `POST /api/push/subscribe` — enregistre l'abonnement (upsert sur `endpoint`).
- `GET /api/cron/notify` — **protégée** (vérifie `Authorization: Bearer ${CRON_SECRET}`). `export const runtime = 'nodejs'`. Client **admin** : sélectionne les `planned_sessions` où `date = aujourd'hui (Europe/Paris)`, joint les `push_subscriptions` de chaque `user_id`, envoie via `web-push`. Les abonnements morts (statut 404/410) sont supprimés.

### 5.8 Planificateur
`vercel.json` :
```json
{ "crons": [{ "path": "/api/cron/notify", "schedule": "0 6 * * *" }] }
```
`0 6 * * *` UTC ≈ **08:00–08:59 Europe/Paris en été (CEST)**, 07:00–07:59 en hiver (CET). Dérive saisonnière documentée et acceptable ; affinable plus tard.

### 5.9 Client admin Supabase
`app/lib/supabase-server.ts` : ajouter `createSupabaseAdmin()` utilisant `SUPABASE_SECRET_KEY` (sans token, bypass RLS). Utilisé **uniquement** par `/api/cron/notify`.

## 6. Dépendances
- `web-push` (npm) — génération VAPID + envoi (runtime Node, pas Edge).

## 7. Cas limites / erreurs
- **Permission refusée / non supportée** : plan fonctionnel, pas de notif. Aucune relance intrusive.
- **iOS non installé** : indice d'installation au lieu de la demande de permission.
- **Pas d'abonnement pour un user** : le cron l'ignore.
- **Abonnement expiré** (404/410) : supprimé côté cron.
- **Aucune séance prévue ce jour** : le cron n'envoie rien à ce user ; l'`IdleScreen` garde son CTA standard.
- **Re-saisie d'un jour** : upsert via `UNIQUE(user_id, date)`.

## 8. Tests (vitest)
- Helpers de date/semaine (lundi, `isoDate`, « aujourd'hui Europe/Paris »).
- `buildStatusItems(plansToday, alerts, greeting)` — fonction pure : greeting retiré si séance du jour ; ordre alertes → planif.
- `selectNotifiable(plannedRows, subscriptionRows)` — fonction pure de jointure du cron (qui notifier, quel message), testée sans réseau.
- `planDiff(weekStart, existing, edited)` — calcule upserts + deletes pour le batch.

## 9. Prérequis (action utilisateur)
1. Générer les clés VAPID : `npx web-push generate-vapid-keys`.
2. Ajouter en env (local `.env` + dashboard Vercel) :
   - `NEXT_PUBLIC_VAPID_PUBLIC_KEY`, `VAPID_PRIVATE_KEY`, `VAPID_SUBJECT` (`mailto:…`)
   - `SUPABASE_SECRET_KEY` (clé secrète `sb_secret_…`, côté serveur uniquement)
   - `CRON_SECRET` (chaîne aléatoire ; Vercel l'injecte dans l'en-tête du cron)
3. Appliquer la migration SQL dans Supabase.
4. Déployer sur Vercel avec `vercel.json` (cron activé sur le projet).

## 10. Découpage d'implémentation (aperçu)
1. Migration SQL + `usePlan` + routes `/api/plan` (GET/POST) → plan persistant.
2. `PlanWeekModal` + chip `StatusSpot` (sans push) → planifier/voir.
3. Pré-sélection jour J : `NavContext.plannedType` + split-button `IdleScreen`.
4. Push : `web-push`, env/VAPID, `/api/push/subscribe`, `sw.js`, `push.ts`, abonnement après 1er save.
5. Cron : `createSupabaseAdmin`, `/api/cron/notify`, `vercel.json`.
6. Tests des fonctions pures.
